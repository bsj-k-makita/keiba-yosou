# 実装計画書

`言語定義.md` と現行 `src/` を前提に、次の5テーマの計画をまとめる。計画書を磨くだけで終わらないよう、**責務の境界**と**消し候補の扱い**、**不要文の判断基準**、**作業順**を最初に固定する。

---

## 責務の固定（実装前に必読）

ラベルが並ぶと「全部同じ意味」に死ぬので、次の3つを**分離**する。

- **今回向き** ＝ **能力面**の適合（能力 × 今回求められる重み。算出は `fitScore` 系）。
- **展開適合** ＝ **位置取り面**の適合（脚質 × ペース・バイアス等の展開条件）。
- **買い判断** ＝ **能力面 + 位置取り面 + 評価（順位・補正差分など）**を統合した**最終ラベル**。

```text
今回向き   = 能力 × 今回重み
展開適合   = 脚質 × ペース・バイアス
```

今回向きは能力面の適合、展開適合は位置取り面の適合として扱い、**両者は統合せず、別ラベル**として常に出し分ける。

---

## 表示責務マトリクス

| 表示項目 | 意味 | 算出元 | 表示場所（目安） |
|----------|------|--------|------------------|
| 補正評価 | 今回条件込みの総合評価（絶対点） | 能力 × 最終重み → `adjustedScore` | カード上段 |
| 最終評価 | メンバー内比較＋展開 | 相対化＋展開ボーナス → `finalEvaluationScore`／`finalRank` | 詳細・印・ソート基準 |
| 今回向き | 能力と今回重みの一致度 | `fitScore` → 高/中/低 | カード中段 |
| 展開適合 | 脚質と展開の噛み合い | `runningStyle` × `pace` / `bias` 等 | 脚質バッジ付近（横） |
| 買い判断 | 最終判断ラベル | 順位・今回向き・展開適合・他ルール | カード上段 |
| 消し | 条件不一致が強い馬 | 下記「消し候補」のルール | 結論 / カード |

マトリクスに無い雑多な文言を同じ帯域に積まない。増えるのは**意味が違う項目だけ**に限定する。

---

## 消し候補の判定（横断）

**消し候補**は、例えば次の**いずれか**を満たす、など単一条件では**即「消し」**にしない。

- 補正後順位が**下位**
- **今回向き**が**低**
- **展開適合**が**×**
- 補正**差分**が大きく**マイナス**

**原則：2条件以上**を満たした場合に**消し候補**とする。一要素だけで消すと雑になるため、人間の慢心（と実装の暴発）に備える。具体的閾値は `markAssigner` / `buyLabel` 系で定数化し、レビュー可能にする。

## 1. 脚質バッジ追加

### 目的

`言語定義.md` のカード上段「印 / 馬名 / 脚質」を満たし、一覧で走法が一目で分かるようにする。

### 脚質と能力（必ず分離）

脚質は**位置取り傾向**であり、**能力値（スピード・末脚等）とは別管理**とする。

- **誤禁止**：「先行だからスピード高い」「差しだから末脚高い」のように、脚質から**能力スコアを直接**変化させること。
- **正**：脚質は**データ属性**＋**展開適合**の入力に使う。能力5軸は従来どおり固定プロフィール（言語定義の「能力は固定」方針）。

```text
脚質から能力値を直接変更しない。
ただし、展開適合の判定には使用する。
```

### スコープ

- 脚質の型定義、データ（各馬1値）、`HorseEvaluationCard` 上段へのバッジ表示（タブレット幅でも破綻しにくい短い表示）。

### 非スコープ

- 脚質を根拠に**能力値の補正**（スコア乗数を脚質で変える等）。重み補正は**条件**側（`RaceCondition`）の責務とし、脚質は**位置取り**レーンに閉じる。／展開予測の詳細物理シミュ（別タスク可）。

### 前提

- 現状 `HorseAbility` に未搭載。`言語定義` §4 の7値（逃げ・先行・好位・差し・追込・自在）に相当する **固定 union 型**に落とす。

### 実装方針

| 分類 | 内容 |
|------|------|
| 型 | `abilityTypes.ts` または `runningStyle.ts` 新設で `RunningStyle` と日本語短表記 map（4-A の lingo から参照可能に） |
| データ | `data/mockRace.ts` の `SAMPLE_HORSES` に `runningStyle` を追加（将来 API/CSV は同形で差し替え） |
| UI | `HorseEvaluationCard` ヘッダ：馬名の隣 or 下にバッジ。展開適合は**脚質横**に配置可能 |
| スタイル | `index.css`：既存 `horse-card` トーンに合わせた小バッジ |

### 受け入れ基準

- 全サンプル馬に脚質が表示され、未定義の馬がない（または明確なフォールバック規則に従う）。
- スクリーンリーダ向けに、ラベルまたは `aria` で「脚質：先行」など意味が通る。

### リスク・メモ

- データに脚質が常に無い場合の扱い（非表示 / 「未設定」）を事前に決める。

## 2. 展開適合ラベル追加

### 目的

`言語定義` §5（ペース・バイアス）および **§6 展開適合**（`◎` `○` `△` `×`）を、**今回向き（高/中/低）** とは区別して表示する。

### 今回向きと展開適合（責務明確化）

- **今回向き** ＝ **能力** × 今回求められる**重み**（能力面）。
- **展開適合** ＝ **脚質** × ペース・バイアス等の**展開**（位置取り面）。

両者は**役割が近いので事故りやすい**が、**統合せず、別ラベル**で表示する。曖昧な併記（「なんとなく合う」）を避ける。

### スコープ

- 馬の脚質（1）と今回の展開想定（`RaceCondition` の `pace` / `bias` を主因）の**適合**を、1記号＋極短い「展開適合」系ラベルで示す。

### 非スコープ

- 出馬表の印（`HorseScoreResult.mark` の `◎` `○` 等）と**同一の意味**にしない。UI で**展開適合**と明示。
- 今回向き用の `fitLevel` との**1本化**。

### 前提

- 現状の「今回向き」は `fitScore.ts` の `computeFitScore` ＋ `fitLevelFromScore`。**展開適合**は**別関数**：`computePaceFit( runningStyle, condition ) → "◎" | "○" | "△" | "×"` 等。ルールは**定数表**＋コメントで説明可能に。

### 実装方針

| 分類 | 内容 |
|------|------|
| ドメイン | 例: `paceFit.ts` 。`pace` / `bias` ＋ `RunningStyle` のルール表 |
| UI | 新コンポーネント。`FitLabel` とは**別**に「展開適合：◎」。**脚質横**（マトリクス通り） |
| データ | 1 に依存。脚質未導入時はデモ用の制約を文書化 |

### 受け入れ基準

- 条件で `pace` / `bias` を変えると、展開適合が一貫して変化する。
- 「高/中/低の今回向き」と併せて、**曖昧な1文**に落ちない（買い判断の節を参照）。

### リスク・メモ

- 根拠不足は不信感。必要なら**1行**だけ根拠（展開用）を併記し、5 の「向き用」と混ぜない。

## 3. 不要テキスト削除

### 目的

`言語定義` §15：抽象表現・**全馬同文**・意味のない文章を削ぎ、判断に直結する。

### カード常時表示に**残してよい**文（目安）

- その馬**固有**の**強み**
- **今回条件**との**一致点**（向き or 展開、どちらのレーンか明示）
- 評価が**上がった / 下がった理由**（補正差分に紐づく）
- **買い判断**に**直結する**注意点

### **削除**しやすい文（対象候補）

- **全馬に共通**する説明
- 条件名を**言い換えただけ**の説明
- **スコア算出**の**一般**説明（学習用コピー）
- 抽象的な**安心文**（何を買うか決めない）

上記を実装前に**チェックリスト**化し、各コンポーネントへ割り当てる。ブレたら「常時 or 折りたたみ」の二択で揃える。

### スコープ

- カード：`HorseEvaluationCard`（`理由：` 行、条件由来の長文、詳細 `result.reason` の出し方）。
- アプリ枠：`App` ヒーロー、`RaceEvaluationSummary` / `RaceConclusionPanel` の**重複**条件説明。
- `reasonGenerator.ts` の `generateScoreReason`：折りたたみ内短文化 or 要約1行＋詳細分離。

### 非スコープ

- 5 の「向き専用」補足の**文字数**調整。ここでは**重複**と**常時可視のノイズ**を主に削る。

### 実装方針

- 「削除 / 1行 / 折りたたみのみ」に**必ず**振り分け。
- `buildEvaluationCardSummaryLine`・`buildFitSupplementLine`・ヒーローが**同趣旨**なら 1 系統へ。

### 受け入れ基準

- 同趣旨の説明が1画面内で2回以上続かない（意図的再掲を除く）。
- 上記**削除候補**のパターンが、常時に残っていない。

### リスク・メモ

- 短かしすぎ → 詳細へ退避。

## 4. 言語定義.mdを実装に反映

### 目的

`言語定義.md` を**単一の参照元**に近づけ、用語のブレを防ぐ。

### スコープ

- 用語定数（能力5軸、脚質7、ペース3、バイアス、展開適合4段、今回向き3段、補正名）。
- 既存 `ABILITY_LABELS`・`adjustments` 系の **1系統化**（`lingoConstants.ts` 等）。
- `abilityGrades.ts` と §3 の整合。§7 コース特性と `getBaseWeights` / `venue`。
- §8 補正の優先（`言語定義`: コース ＜ 馬場 ＜ 展開）の実装レビュー、必要なら**最小**修正。

### 非スコープ

- `言語定義.md` 本文の書き換え（依頼がなければ**コード**が主）。

### 実装方針（推奨フェーズ）

| フェーズ | 内容 |
|----------|------|
| **4-A** | **最初に**実施。定数・型集約。脚質・展開適合・向きの**全ラベル**がここに依存可能に |
| 4-B | コンポーネントと生成文を**段階的**に定数参照へ |
| 4-C | `言語定義.md` と1:1チェックリスト（レビュー用） |

**4-A を最初にする理由**：脚質・展開適合・今回向きは**用語定数**に全て依存する。ここを後回しにすると、実装のあとに文言だけで転ぶ。

### 受け入れ基準

- 主要ラベルは lingo 経由で一貫。
- §16 NG 定義に**触れる**文がデフォルトで出ない。

### リスク・メモ

- 一括置換の爆発 → **4-A を小さく**マージし、4-B を分割。

## 5. 今回向きの理由を短文化

### 目的

§9「今回向き」と §15（理由1行）に合わせ、**能力面**の補足だけを**短文**に閉じる（展開は 2 へ、脚質はバッジへ）。

### スコープ

- `buildFitSupplementLine`：全馬同文の解消。`(horse, condition, finalW)` 等へ。`fitScore` 上位能力と**辻褄**を合わせる。
- `FitLabel` の補足：`line-clamp` 等で物理的制限。

### 非スコープ

- 3 の**重複削除**専用タスク。ここは**向き＝能力面**の一行。

### 実装方針

- **能力1〜2語＋1動詞**目安。展開の説明を**足さない**（位置取りは展開適合の仕事）。

### 受け入れ基準

- 補足が**約40〜60文字**目安で1行。`高/中/低` と矛盾しない。

### リスク・メモ

- 「向き中・展開○・相手」で**曖昧**に見えたら、**買い判断**（順位＋二面）の説明を`RaceConclusionPanel` 等に寄せ、カード上段で**一貫**させる（責務の固定参照）。

## 6. レース内相対化・最終評価スコア（実装済み・拡張可）

### 目的

`docs/race-evaluation-spec.md` の「評価軸は材料、最終比較は同レース内で相対化」をコードに落とす。絶対点の補正後スコアに加え、**出走表内 min-max 正規化（0〜100）** と **展開適合ボーナス** から `finalEvaluationScore` を算出し、**印・買い判断・一覧ソートの基準順は `finalRank`** とする。

### 実装状況（ベースライン）

| 項目 | 内容 |
|------|------|
| ドメイン | `finalScoring.ts`：`computeRaceRelativeScores`、`paceFitToBonus`、`combineFinalEvaluationScore` |
| パイプライン | `evaluateRace`：加重スコア算出 → 相対化 → 展開ボーナス → `finalRank` → 印 / 消し / 買い |
| 型 | `HorseScoreResult` に `raceRelativeScore` / `paceFitBonus` / `finalEvaluationScore` / `finalRank` |
| JSON 出力 | `EntryEvaluationBlock` に上記3数値を含む（`buildEvaluationData`・`convertToRaceEvaluationData` 保存パス） |
| UI | 詳細に「レース内相対」「最終評価」、順位変動は **標準 → 最終順** |

### 非スコープ（次の改善候補）

- 相対化を min-max 以外（z-score、順位百分位）に差し替える検証
- 展開ボーナス幅のデータドリブン調整（現状は `PACE_FIT` ごとの固定値）
- 表示マトリクスの「補正評価」の文言を「素点」寄りと「最終」寄りで二段表示にするかの UI 判断

### 受け入れ基準

- `npm run build` が通る
- 条件変更で相対・最終が同一レース内で一貫して変化する
- 旧 JSON / 分析ルートでも `evaluateRace` 再計算で破綻しない

## 横断：レビュー後の推奨実装順

```text
1. 4-A 用語定数化
2. 1 脚質型・データ・バッジ
3. 本書「表示責務マトリクス」の通り、配置・凡例の固定（足りなければ UI タスクとして明示）
4. 2 展開適合ラベル
5. 3 不要テキスト削除
6. 5 今回向き短文化
7. 6 レース内相対化（本節・ベースライン完了）
```

図式：

```text
4-A → 1 → 2 → 3 or 5（3 と 5 は重複に注意しつつ並行可）
```

旧来の「脚質から先、4-A 並行」は**採用しない**（4-A 先行に統一）。
