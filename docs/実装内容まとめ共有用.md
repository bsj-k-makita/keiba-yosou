# 馬別評価：フロント表示とバックエンドロジック対応表

このドキュメントは、現状の UI に「何が」「どこから」出ているかを整理し、`evaluateRace` を中心とした評価パイプラインと突き合わせたものです。設計見直しのたたき台として使ってください。

**主なソース**

- UI: `src/components/race/HorseEvaluationCard.tsx`, `HorseListTable.tsx`, `RaceBetPanel.tsx`, `RaceConclusionPanel.tsx`, `RaceEvaluationSummary.tsx`
- 評価コア: `src/domain/race-evaluation/scoreCalculator.ts`（`evaluateRace`）
- 型: `src/domain/race-evaluation/abilityTypes.ts`（`HorseScoreResult`, `HorseAbility`, `RaceCondition`）
- ViewModel（期待値・加重レーダー）: `src/viewModel/raceEvaluationViewModel.ts`, `src/lib/pipeline/evaluationPipeline.ts`
- 関連既存資料: `docs/EvaluationLogic.md`, `docs/race-evaluation-spec.md`, `言語定義.md`

---

## 1. データの流れ（概要）

1. レース JSON（例: `RaceEvaluationData`）から `HorseAbility` 配列へ変換（`src/lib/race-data/` 周辺）。
2. `runRaceEvaluationPipeline(horses, condition)`（`evaluationPipeline.ts`）が内部的に `evaluateRace` を実行。
3. 各馬の `HorseScoreResult` と、オッズ・ソフトマックス用の `RaceEvaluationViewModel` が UI に渡る。
4. 一覧は `finalRank` 等でソートされた `HorseScoreResult[]` を前提に描画（`RaceDetailView` 等）。

## 2. 馬カード（`HorseEvaluationCard`）— 表示項目とロジック

| 表示 | ソース | ロジックの要点 |
|------|--------|----------------|
| 印（◎○▲△☆） | `result.mark` | `markAssigner.ts`: `finalRank` ベースで初期印（◎1位/○2位/▲3位/△4-5位、条件成立で☆）。その後 `applyStabilityRescueMarks`（安定度救済）→ `fillRequiredMarks`（必須印補完）→ `assignHokkakeRoles`（△1安定/△2物理/△3狙い）→ `ensureTriangleMarks(results,4,dismissIds)`（△を4頭に拡張）。 |
| 馬番・枠色 | `gate`, `horse.frameNumber` | 見た目のみ。評価式には直接効かない。 |
| 馬名・脚質 | `horse.horseName`, `horse.runningStyle` | 生データ。 |
| 順位変動バッジ（↑↓） | `baseRank`, `finalRank` / `adjustedRank` | `scoreCalculator` の `assignRanksForScore`。補正前後の差。 |
| **頭／軸** バッジ | `result.roleHint`, `varianceScore` | `maxPerformance.ts` の過去走着順スコアの標準偏差。高い→「頭」、低い→「軸」、データ不足→「判定不能」。 |
| レーダー（形状） | 既定は `viewModel.weightedRadar`、無ければ生5軸 | ViewModel: 最終ウェイトで各軸を掛け、最大値で正規化し 0〜100。キャプションどおり「補正後能力バランス」は**条件ウェイト込みの見た目**。 |
| レーダー下1行タイプ | `inferRadarShape(horse)` | 5軸のばらつき・上位2軸差で「バランス型」「スピード型」など（`radarShape.ts`）。幾何学名ではなく言語定義寄りラベル。 |
| 5軸数値・等級 pills | `horse[k]`, `grades[k]` | 等級は出走頭数内順位から S〜C（`abilityGrades.ts` の `computeAbilityLetterGrades`）。 |
| **評価スコア（調整後）** | `result.adjustedScore` | `calcHorseScore(horse, finalWeights)`。最終正規化ウェイトでの加重能力。 |
| **基礎能力** / **条件適性差** | `intrinsicAbilityScore`, `conditionFitDelta` | 基礎＝`baseAbilityCore`＋再現性−大敗＋`enginePeakBonus`（第1層）。条件適性差＝`adjustedScore − intrinsic`。 |
| スコア変化インジケータ | `result.scoreDiff` | `adjustedScore − baseScore`。±2 が「大幅」（`ScoreDiffIndicator.tsx`）。 |
| **補正前 → 補正後（±Δ）** | `evaluationBaselineScore`, `finalEvaluationScore`, `evaluationAdjustmentDelta` | `capAptitudeSwing` で相対スコアからの振れ幅を制限。baseline は弱めの conditionImpact／final はフル強度＋extras。 |
| **買い** | `result.buyLabel` | `buyLabel.ts`＋消しセット。下記§5。 |
| 市場アラート短文 | `computeMarketAlertLabel` | `evaluationTags.ts`。オッズ順位・`buyLabel`/順位・ラップ計・文脈計から「危険な人気馬」「大穴候補」。 |
| クイック補正バッジ | `adjustmentBadges` | 「ラップ適合」「前走不利解消」「バイアス逆行救済」「耐性バフ」「鞍上強化理由」「馬格/耐久加点」等（`scoreCalculator`）。 |
| コース特性一致 | `courseTraitBonus`, `courseTraitReasons` | `courseTraitResolver`＋`structuralPhysicalHits`。ボーナス>0 で 🧭 バッジ。 |
| 鞍上強化バッジ | `jockeyRiderBonus` | `jockeyContext.ts`。Tier昇格/舞台巧者/厩舎シナジー/継続騎乗の合算。 |
| △役割バッジ（[安定][物理][展開]） | `hokkakeRole` | `assignHokkakeRoles`。△候補の性格付けを表示。 |
| 陣営・気性バッジ | `computeConnectionSpecialBadges` | 騎手/厩舎コース勝率、長距離+複勝、`temperamentRisk` 等。 |
| EV 強調スタイル | `viewModel.effectiveEv` | >1.25 で `horse-card--ev-gold`。ViewModel で算出。 |
| 短評（折りたたみ上の1行） | `finalEvaluationScore`, `buyLabel`, 印 | 集約表示。 |
| **今回向き** | `FitLabel` | `computeFitScore`＋`fitLevelFromScore`（0.72以上「高」等）。説明文は `buildFitSupplementLine`。 |
| 強み行 | `strongAbilities`＋grades | `strongAbilities.ts` の抽出結果。 |
| 能力バー | `AbilityBar` | 同カード詳細内。 |
| **レース相対・最終評価・内訳** | `raceRelativeScore`, `finalEvaluationScore`, 各 bonus | メタ説明は UI の `title` に集約。**ラップ**は `lapShapeFitBonus + raceAnalysisBonus + lapSustainBonus + lapQualityBonus`（耐性バフは lapStack 内で cap）。**文脈**は血統・枠・枠×脚質・陣営・傾向・前後傾・不利恩恵。 |
| 素ブレンド・合成素点 | `baseAbilityCore`, `raceAdjustedInput` | `abilityCoreScoring.ts`。合成素点＝内在＋条件スコア＋MAX性能＋クラス/ステップ＋強度係数。 |
| 過去走シグナル | `pastRunInsight` | `pastRunDerivedSignals.ts`。 |
| 基礎順位変化 | `baseRank → orderRank` | 同上トグル内。 |
| 同タイプ馬 | `TypeMatchList` | `findL1CloseTypePeers`（能力5軸 L1）。 |
| **詳細評価**（pre本文） | `result.reason` | `generateScoreReason`（順位変化・馬場・バイアス・ペース・ウェイト増減との整合を短文化）＋数値内訳1行（最大260文字トリム）。 |

### 馬カードで「最終順位」の基準

- **並び順・印・買いラベル**に使われる順位キーは原則 `finalEvaluationScore` の **`finalRank`**（`scoreCalculator.assignRanksForScore`）。

## 5. `evaluateRace` の処理順（裏側ロジック要約）

実装は `scoreCalculator.evaluateRace`。概略のみ。

1. **能力ブレンド**  
   `blendAbilityWithPastRuns` で過去走を反映した仮想能力へ。

2. **レース横断コンテキスト**  
   `resolveEffectiveRacePace`（pace補完）→ `inferPaceSeverityKind`（high/slow/neutral 推定）→ `computeStyleSignalFactor`（脚質分散）→ `computeKickInTopFractionMap`、`computePaceScenarioAmplifier`。  
   ウェイトは馬ごとに `getFinalWeightsForHorse` を使い、`paceSeverity` と脚質で動的増幅。

3. **各馬ベース結果（第1〜3層の素）**  
   - `rawBase` / `rawAdj`（標準ウェイト／最終ウェイト）  
   - `intrinsicAbilityScore` = `baseAbilityCore + reproducibility − risk + enginePeakBonus`  
   - `raceAdjustedInput`（相対化前の合成素点）  
   - ラップ: `computeLapShapeFit`, `computeRaceAnalysisBonus`, `computeStaminaResilienceBonus`, `todayLapKind`

4. **レース内相対化**  
   `computeRaceRelativeScores(raceAdjustedInput)`。  
   `adjustmentStrength === "strong"` のとき **`absolute_delta`** モード、それ以外 **`normalized`**（z-score または分散小時の min-max）。

5. **最終評価スコアの合成**  
   `combineFinalEvaluationScore` が核:  
   `relative + paceFitBonus + lapStack + distance + classCombined − variancePenalty + contextual + conditionImpact`。  
   さらに `courseTrait`、`lastRunReset`、`bias disadvantage recovery`、`capAptitudeSwing` で **能力主導とのバランス**を制御。  
   `contextual` には血統/枠/枠脚質/陣営/傾向/前後傾/不利恩恵に加え、`jockeyRiderBonus`・`heavyWeightPowerBonus`・`staminaTestBonus` を加算。

6. **順位**  
   `baseScore` → `adjustedScore` → `finalEvaluationScore` の3段でランク。

7. **第4層オッズ歪み（結果オブジェクト側）**  
   `detectOddsDistortion`（能力パーセンタイル・近走ミスマッチ・割安っぽさ）。`HorseScoreResult` に `oddsDistortion*` を格納。

8. **印・買い判定の前後処理**  
   `assignMarks` → `applyLongshotGuard`（50倍以上・大敗・素点5位外などで ◎○→▲ 降格）→ `collectDismissIds` → `assignBuyLabels`。  
   ここで一度 `buyLabel===DISMISS` は印をクリアするが、後段で安定救済/△拡張が入る。

9. **買いラベル**  
    `assignBuyLabels`:  
    - 1〜3位: 本命候補 / 対抗 / 単穴  
    - 4〜5位: `scoreDiff≥3` かつ **`oddsDistortionFlag` 相当**でのみ「穴候補」、否則「相手」  
    - 6位以下: 消し  
    - **本命ブロック**: `shouldBlockHondeCandidate`（直近5走で10着以下が3回以上かつ good走0）。

10. **印の仕上げ**  
    `applyStabilityRescueMarks`（安定度上位の○/▲救済）→ `fillRequiredMarks`（◎○▲☆△の不足補完）→ `assignHokkakeRoles`（△1/2/3）→ `ensureTriangleMarks(results,4,dismissIds)`（△を4頭へ拡張）。  
    最終安全策で **`dismissIds` に入る構造消し馬だけ** 印・役割を再クリア。

11. **理由テキスト**  
    `generateScoreReason`。

## 5.1 加点ロジック一覧（現状実装）

### A. 主スコア合成に直接入る加点

- `paceFitBonus`（`computePaceFitEvaluation`）  
- `distanceFitBonus`（`computeDistanceFitBonus`。距離変更バイアス含む）  
- `classLevelBonus + stepPatternBonus`（クラス・ステップ）  
- `lapShapeFitBonus + raceAnalysisBonus + lapSustainBonus + lapQualityBonus + staminaResilienceBonus`（上限整理あり）  
- `contextualBonuses`（血統/枠/枠脚質/陣営/傾向/前後傾/不利恩恵）  
- `conditionImpactBonus`（`scoreDiff` と `adjustmentStrength` 依存）

### B. 追加の文脈加点（contextual に統合）

- `jockeyRiderBonus`（`computeJockeyRiderBonuses`）
  - Tier昇格
  - 舞台巧者
  - 厩舎シナジー
  - 継続騎乗
- `heavyWeightPowerBonus`（馬格×馬場/北海道系）
- `staminaTestBonus`（坂・タフ条件への耐久適性）

### C. 直前/安全装置系の加点

- `lastRunResetBonus`（クイック補正）
- `lapFocusBonus`（L2指標とコース要求一致）
- `BIAS_DISADVANTAGE_RECOVERY_BONUS`（前走バイアス逆行救済）
- `courseTraitBonus`（コース物理特性一致）
- `capAptitudeSwing`（過剰逆転を抑える振れ幅制御）

## 6. ViewModel（`RaceEvaluationViewModel`）— カード/table と期待値

`buildRaceEvaluationViewModel` は `evaluateRace` **後**に、`finalEvaluationScore` から softmax 確率（`evaluationPipeline.softmaxDistribution`）を作る。

| 項目 | 意味 |
|------|------|
| `weightedRadar` | 上記。UI レーダーの既定値。 |
| `baseAdjustedWinProbability` | 歪み前のソフトマックス確率。 |
| `adjustedWinProbability` | `detectOddsDistortion` の `probabilityBoost` を乗算（上限 0.95）。 |
| `effectiveEv` | `adjustedP × odds − evMargin`（16頭以上でマージンやや大）。オッズは `investment.actualOdds` 優先、`signals.winOdds` 次。 |
| `kellyFraction` | 標準ケリー、上限 0.4。 |
| `evHot` | EV > 1.25 |
| `layerBreakdown` | UI コンパクト用: `enginePeakBonus`, stamina resilience フラグ/強度, `todayLapKind`, `staminaResilienceBonus` |

※ ViewModel と `HorseScoreResult` の第4層フィールドは同じ `detectOddsDistortion` だが、**確率ブーストを使うのは ViewModel 側**（結果側はフラグ／スコア／理由）。

## 8. `RaceCondition` が効く主な項目（ユーザー操作/UI連動）

- **馬場・バイアス・ペース・直線快慢** → ウェイト調整（`adjustments.ts`）
- **`abilityFocus`** → 選択軸 3×して再正規化
- **`abilityPriority`** → プリセット 1.5×（stamina は sustain もセット）
- **`adjustmentStrength`** → Impact 係数・相対モード（strong で absolute_delta）
- **`trackCushion01`, `turnCount`, surface, venue, distance, ...** → `venuePhysicalFactors`, 距離適性、ペース適性等
- **`section200mSec`, `raceAnalysis`** → ラップ形状・蓄積分析ボーナス
- **`quickAdjustments`** → 前走不利リセットなど
- **`softmaxTemperature`** → 期待値 softmax（`evaluationPipeline`）

## 10. 設計精査時の注意点（実装から見えるギャップ）

- **一覧の「スコア」列**は `adjustedScore` であり、**カードの「最終評価」や結論の順位**は `finalEvaluationScore` ベース。ユーザーが混同しやすい。
- **レーダー**は ViewModel 時は「重み付き正規化形状」、無いときは生5軸。説明文は前者を想定。
- **短評**は `investment` があると **投資用テンプレに完全切替**（構造化短評は使わない）。
- **消し馬**でも数値（`finalEvaluationScore` 等）は計算済みのまま残る。  
  なお現在は `ensureTriangleMarks` の導入により、**構造消し（dismissIds）でなければ `buyLabel=消し` でも△が付く場合がある**。
