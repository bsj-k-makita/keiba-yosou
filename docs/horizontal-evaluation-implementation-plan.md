# 横比較精度改善 実施計画書

## 背景
- 現状の評価は、同一レース内の条件適合と相対化に強い一方で、レースレベル（G1/G2/G3 等）の実績が十分に反映されない。
- そのため、実績上の強豪馬が横比較で過小評価されるケースが発生し得る。

## 目的
- 「どのレベルのレースで、どの内容で走ったか」を評価へ組み込み、横比較の妥当性を改善する。
- 既存UI/スコアを壊さず、段階的に導入する。

## 対象範囲
- 取り込み: `scripts/lib/parseNetkeibaPastRuns.mjs`
- ドメイン評価: `src/domain/race-evaluation/*`
- 型/変換: `src/lib/race-data/*`, `src/domain/race-evaluation/*`
- 検証: 代表レース（例: `202608030411`）で順位差分確認

## 実装方針（段階導入）
### Phase 1（今回実装）
- 過去走に `raceName` を保持し、名称からレースクラス（G1/G2/G3/OP/3勝/2勝/1勝/新馬/未勝利）を推定。
- レースクラスと走破内容（着順・着差）を合成した `classLevelBonus` を算出。
- `finalEvaluationScore` に `classLevelBonus` を加算する。
- 出力データに `classLevelBonus` を保持し、検証しやすくする。

### Phase 2（次段）
- raceId を使って同一レース出走馬の事後実績を集約し、`fieldStrength`（相手レベル）を推定。
- `classLevelBonus` を `fieldStrength` で補正して更新。

### Phase 3（次段）
- タイム・上がり・馬場差補正の導入。
- 速度指標を「同条件比較可能な指数」に正規化。

## 受け入れ条件
- `npm run build` が成功する。
- 既存評価が破綻しない（スコア範囲 0-100 維持）。
- `classLevelBonus` が各馬に出力される。
- G1 実績馬の極端な過小評価が緩和される（サンプルレースで確認）。

## リスクと対策
- レース名ベース分類の誤判定: 正規表現を保守しやすい関数に集約する。
- データ欠損（raceName無し）: 中立（bonus=0）フォールバック。
- 過剰補正: ボーナス範囲をクランプし、比率を調整可能にする。

## ロールアウト
- まず Phase 1 を反映して差分確認。
- 問題なければ既存データの再取得・再計算を段階的に実施。
